# v2.2.2 — Website Designer Actually Speaks Every Language — What Changed

You reported that the Website Designer tool's package contents stayed
in English no matter which language you picked, and separately that
the language switcher disappears entirely in certain languages. You
asked me to ensure translation integrity across the whole site.

## Root cause: the Website Designer tool was never wired into i18n at all

Every other translated page on this site is static HTML with
`data-i18n` attributes that `js/i18n.js` swaps at load time. The
Website Designer tool doesn't work that way — its content (catalog
feature titles, category names, the "Included in your X package"
summary, the live price ticker, the cost breakdown, the illustrative
preview cards) is all generated by `js/website-designer.js` reading
from `starter-catalog.json`/`business-catalog.json` and building
strings in JavaScript. None of that had ever been connected to the
translation system — not a regression, it was simply never built.

## The fix: a translation bridge for JS-generated content

- `js/i18n.js` now exposes `window.LTS_I18N` — a small API
  (`t(path, fallback)`) that looks up the active language's dictionary
  and falls back to English if a key is missing, plus an
  `lts:langchange` event fired whenever the visitor switches language.
- `js/website-designer.js` now routes every piece of dynamically
  generated text through that lookup: catalog item/category names, the
  included-package summary, bundle/savings messaging, the cost
  breakdown, form status and error messages, and the illustrative
  preview card text — and listens for `lts:langchange` to re-render
  already-built content (category panels, the live preview, badges) if
  you switch language mid-session, restoring your checked features
  afterward rather than losing your selections.
- Also added `data-i18n` to the Step 3 business-brief fields from the
  last release (description, industry, service area, services list,
  staff/testimonials/FAQ/blog/etc.) and to a few small static strings on
  this page that had silently never gotten `data-i18n` in the original
  rollout ("Your estimate," the American Heroes Discount checkbox text,
  the empty-preview placeholder).
- Translated roughly 200 new strings — 77 catalog feature names, 22
  category names, ~65 dynamic UI strings, and ~40 form field labels —
  into all 15 non-English languages, matching the tone and terminology
  already established in each language's existing translation file.

## Second bug found while checking languages, unrelated to the above

The Website Designer form's anti-spam honeypot field used
`position:absolute; left:-9999px` to hide itself — the same
left-to-right-only trick that caused the RTL blank-space bug fixed for
the "skip to content" link in v2.2.1, and apparently the only other
place on the site still using it. Switched it to the same
`.visually-hidden` clip-based technique every other form on the site
already uses, which is direction-safe.

## The "language switcher disappears" bug: not reproduced

Tested this specifically and broadly: 10+ languages screenshotted
directly (English, Spanish, French, Chinese, Arabic, Filipino, German,
Korean, Haitian Creole, Hindi), all 16 checked structurally, desktop
(1280–1680px), tablet (768px), and mobile (375px) widths, RTL layout,
opening/closing the language dropdown itself, and clicking through the
Website Designer's package-selection flow (the page most likely to be
open when this was noticed). The switcher stayed visible and
functional in every case. Could not find a reproduction path — if it's
still happening, the language, page, and browser/device would help
pin it down.

## What's still English-only — scope of this release

This release fixed the Website Designer tool specifically (what was
reported) plus the two bugs above. It did **not** translate the
remaining ~26 pages that were never part of the original v2.1.0
translation rollout (team, portfolio, gallery, blog posts, individual
service pages, booking, etc.) — "ensure translation integrity across
the entirety of the website" is a much larger undertaking (full
marketing copy × 15 languages) than fit in this pass. Flagging this
explicitly rather than letting "fixed the reported bug" read as "the
whole site is now fully translated."
