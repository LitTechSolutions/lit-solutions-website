## 2026-07-14 — Session 16

**Decision:** `netlify/functions/_lib/care_hub_auth.js` bridges TWO legacy Blobs session roles directly to `rbac.js` roles, bypassing `organization_memberships` entirely for both: `"admin"` → `actorRole: "platform_admin"`, and `"staff"` → `actorRole: "technician"`. `authenticateForOrg()` checks both before attempting any membership lookup; `authenticatePlatformAction()` and `authenticateStaffAction()` handle actions with no single owning organization (e.g. `organization.create`, F051's cross-org work queue) the same way.
**Why:** Neither `platform_admin` nor `technician` is in `rbac.js`'s `MEMBERSHIP_BACKED_ROLES` — `platform_admin` is authorized platform-wide, and `technician` is authorized per-*resource* via an explicit `assigned` fact (`ticketWorkflowStore.getAssignedTechnician()`), never via org membership; `rbac.js`'s org-scope check substitutes the `assigned` check for technician instead of an org match. `organization_memberships.organization_id` is also `NOT NULL`, so neither role has a tenant-scoped row it could hold anyway. Both legacy roles already exist and are already used exactly this way by the pre-Care-Hub endpoints (`admin-images.js`, `content.js`, `documents.js`, `messages.js`, `notifications.js` all gate on `session.role === "admin" || session.role === "staff"`), so this reuses an existing, real distinction rather than inventing new infrastructure (a sentinel "platform" pseudo-organization, a second staff/admin table). Originally (earlier in this session) `tickets.js`'s technician flow required an explicit `organization_memberships` row per technician per org; corrected once the `staff` bridge was designed, since a technician's authorization was never actually org-scoped in the first place — the resource-level `assigned` check is sufficient on its own.
**How to apply:** This is the ONLY bridge between the two role systems. The legacy `"customer"` role has zero Care Hub capability on its own and must go through a real `organization_memberships` row (`org_owner`/`org_member`/`read_only_customer`) — do not extend either bypass to `"customer"` without a matching business/authorization decision. Note `authenticateStaffAction()` is intentionally `"staff"`-only, NOT `"admin"`-or-`"staff"` — under `rbac.js`'s current capability map, `platform_admin` has no `ticket.view`/`ticket.work` capability at all, so an admin session cannot use staff-only endpoints like the work queue either. This is a faithful reflection of `rbac.js`'s existing Session 1 design (technicians run day-to-day tickets, platform_admin manages the platform/orgs/billing), not a new restriction introduced here — but it may be worth Dylan's attention for a single-person operation where the one admin account is also doing all the technician work; flagged in `DEV_STATE.json` rather than silently patched.
