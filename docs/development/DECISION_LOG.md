## 2026-07-15 — Session 17

**Decision:** `sql.query(text)` is the correct way to execute an arbitrary DDL string against the live Neon database from a one-off migration script — NOT `sql.unsafe(text)`, despite `unsafe()` sounding like the obvious "run raw SQL" method.
**Why:** `@neondatabase/serverless`'s `sql.unsafe(rawSQL)` returns an `UnsafeRawSql` marker object meant to be interpolated *inside* a tagged-template call (e.g. `` sql`SELECT ... AS ${sql.unsafe(colName)}` ``) — it does not execute a query on its own. Calling and `await`ing it directly resolves immediately to that marker object without ever hitting the database, and (critically) throws no error, so a migration script built this way silently does nothing while reporting success. This was caught immediately by verifying the migration's effects with a follow-up `information_schema` query in the same script (the established pattern since Session 13) rather than trusting the "applied" log line — worth repeating in every future migration script, since this class of silent-no-op is exactly the kind of failure that log output alone won't reveal.
**How to apply:** Every future one-off migration script must use `sql.query(statement)` (one statement per call — a single multi-statement string was also unreliable) and must verify the resulting schema state with a real query afterward, never trust the "no error thrown" signal alone.

**Decision:** Registration is invite-only at launch (`OWNER_DECISIONS.md` #4, resolved by an approved directive rather than a mid-conversation question). Implemented as a genuinely new invitation subsystem (`migrations/002_invitations_and_consent.sql`, `src/policy/invitationLifecycle.js`, `src/db/invitationStore.js`, `netlify/functions/invitations.js` + `invitation-accept.js`), not a re-purposing of the pre-existing `auth-register.js` open-registration endpoint. `auth-register.js` itself was kept, not deleted, but gated behind a new `open_registration` feature flag (F056 settings) that defaults OFF, matching `isFeatureEnabled()`'s existing fail-closed design.
**Why:** Deleting `auth-register.js` would have been a larger, harder-to-reverse change than necessary — the approved directive says open registration should stay "behind a disabled feature flag," implying the code should remain available for a future decision to re-enable it, not be removed. A brand-new invitation subsystem (rather than adapting the registration endpoint into an "invite mode") keeps each endpoint's responsibility singular: `auth-register.js` is "self-service, currently disabled," `invitation-accept.js` is "token-gated, currently the only live path" — and the two can never be confused about which authorization model applies to a given request.

**Why:** The approved directive says "Require email verification," and the invitation link satisfies that requirement by construction: it can only be redeemed by whoever received it at that address, the same security property a dedicated verification email provides. Sending a second verification step after the user already proved receipt by clicking the first link would add friction without adding security. This is the one place this session made a judgment call on an "ordinary implementation ambiguity" per the autonomous-decision rule — documented here, tested (the new Blobs user is created with `verified: true` directly), and reversible (a future session could add a second verification step if this judgment is reconsidered).

**Decision:** The new `queryAuditEvents()` (audit-log endpoint's backing query) uses `sql.query(text, params)` with hand-built parameterized SQL, not the tagged-template form (`` sql`...` ``) used everywhere else in `pgAuditSink.js`.
**Why:** The WHERE clause is genuinely conditional on which filters the caller supplied — a tagged template can't omit a clause based on runtime data, only interpolate values into a fixed clause structure. `sql.query(text, params)` is the same real parameterized-query path the Session 17 migration-runner lesson already established (`sql.unsafe()` is an interpolation marker, not an execution path) — reusing a proven escape hatch rather than inventing a new one (e.g. string-concatenating raw values, which would reopen SQL injection).
**How to apply:** Any future endpoint needing a genuinely conditional WHERE clause should follow this same shape: build a `conditions`/`params` array pair with a small `addCondition()` helper, never string-concatenate a filter value directly into the query text.

**Decision:** `audit-log.js`'s self-audit event (`audit.query`) records flattened, individually-named primitive metadata keys (`filterOrganizationId`, `filterActorId`, etc.) rather than a nested `{ filters: {...} }` object, and records `usedCursor: boolean` instead of the raw opaque cursor string.
**Why:** This was a real bug caught by the live smoke test, not a stylistic choice made up front: `assertValidAuditEvent()` (F008, existing since Session 1) has always required every metadata value to be a primitive (SYS-SEC-012 — "no arbitrary payloads in audit logs"), and the first draft's nested `filters` object violated that, crashing every real query with `metadata.filters must be a string, number, boolean, or null`. None of the 623 unit tests running against fakes caught this, since none of them exercise the real `assertValidAuditEvent` validation path with a nested object — only the live smoke test (real Postgres, real validation) surfaced it. The raw cursor is additionally excluded on purpose, not just to satisfy the primitive-only rule: it encodes a real row's `occurred_at`/`id`, and a log entry about "a query happened" has no reason to also persist a value that could be replayed to resume pagination from that exact position.
**How to apply:** Any future audit event whose natural shape is "here's a filter/options object" must flatten it to named primitive keys before calling `auditRecorder.record()` — there is no metadata escape hatch for structured data, by design (SYS-SEC-012), and unit tests against fake SQL will not catch a violation; only a live smoke test exercising the real `shapeAuditEvent()`/`assertValidAuditEvent()` path will.

**Decision:** TOTP MFA (owner decision #2) was built on `otpauth` (RFC 6238), not a hand-rolled implementation on top of Node's `crypto` module, and not `speakeasy`/`otplib` (the other common choices).
**Why:** The directive explicitly prohibited hand-rolling this with just Node crypto and required reviewing the dependency footprint of whatever library was chosen. `otpauth` has exactly one dependency, `@noble/hashes` — a widely-audited, zero-dependency crypto-primitives library already trusted by security-sensitive projects like `viem` and `noble-curves` — versus `otplib`'s much larger transitive dependency tree or `speakeasy`'s multi-year-stale maintenance status. `otpauth`'s `TOTP.validate()`/`.generate()` also accept an explicit `timestamp` parameter, which is what makes `src/security/totp.js` fully deterministic and unit-testable without mocking `Date.now()` globally.
**How to apply:** `src/security/totp.js` wraps every otpauth call so no other file imports `otpauth` directly — if a future session needs WebAuthn/passkeys (explicitly named as a future direction in the directive), add a parallel `src/security/webauthn.js` rather than overloading this module, keeping each MFA mechanism's library dependency isolated to its own wrapper.

**Decision:** The pre-authentication window between password verification and TOTP/recovery-code verification is a **separate cookie** (`lts_mfa_pending`, 5-minute TTL, HMAC-signed via the existing `sign()`/`verify()` machinery with `type: "mfa_pending"`) — never the real `lts_session` cookie issued early or with a "pending" flag inside it.
**Why:** A single cookie that's sometimes "fully authenticated" and sometimes "password-only, MFA still owed" is a state machine encoded in one value that every consumer would have to remember to check — a much larger attack surface than two cookies with two hard-coded names, where `care_hub_auth.js` and every other Care Hub endpoint only ever reads `lts_session` and gets nothing until MFA actually completes. This also means `tickets.js`/`organizations.js`/etc. needed ZERO changes to enforce "no Care Hub access without completing MFA" — that enforcement falls out for free from the pre-auth token never being a valid `lts_session` value.
**How to apply:** Any future multi-step authentication flow (e.g. WebAuthn as a second factor) should use this same separate-short-lived-cookie pattern rather than smuggling intermediate state into the real session cookie.

**Decision:** `mfa-manage.js`'s "disable" and "reset" actions have byte-for-byte identical effects on stored state (clear the secret, clear recovery codes, force re-enrollment at next login) — the only difference is which audit action name gets recorded (`mfa.disable` vs `mfa.reset`).
**Why:** Because MFA is mandatory for `platform_admin` (not optional), there is no state where "MFA is off" is a valid resting state for this role — the moment either action clears the secret, `auth-login.js`'s existing `enrollmentRequired: !user.mfaEnabled` check forces re-enrollment on the very next login, so building genuinely different mechanics for the two actions would be complexity with no behavioral payoff. The directive listed "disablement, and reset" as separately auditable events, which is satisfied by the distinct audit action names alone — an auditor reviewing the log can still tell whether Dylan said "turn this off" versus "I lost my phone, reset it" from the action name, even though the system did the same thing either way.
**How to apply:** If a future release makes MFA optional for some other role, "disable" (leaves the account MFA-less) and "reset" (forces re-enrollment) would need to genuinely diverge — this identical-effect shortcut is valid only because platform_admin's mandatory-MFA invariant makes them equivalent today.

**Decision:** `json()` (`auth_utils.js`) gained `multiValueHeaders` support for the `Set-Cookie` header specifically, accepting an array of cookie strings instead of just one.
**Why:** `mfa-enroll.js`'s confirm step and `mfa-verify.js` both need to set the real session cookie AND clear the pre-auth cookie in the same response — the first attempt joined both `Set-Cookie` values into one comma-separated header string, which is not valid HTTP (`Set-Cookie`'s own `Expires` attribute legally contains commas, making a comma-joined multi-cookie header ambiguous to parse) and has no precedent anywhere else in this codebase. Netlify Functions run on an AWS-Lambda-proxy-compatible runtime that reads repeated headers from a separate `multiValueHeaders` object, not from array values inside `headers` — so `json()` now splits any array-valued header into `multiValueHeaders` automatically, and every other call site (which only ever passes single string values) is completely unaffected.

**Decision:** Session 20 step 10's TOTP anti-replay fix tracks the absolute 30-second period *counter* that matched (`user.mfaLastUsedCounter`, persisted after every successful verification), not a timestamp, and rejects any future code whose counter isn't strictly greater — implemented as a new `validateTotpToken()` export in `src/security/totp.js` returning `{ valid, counter }`, alongside the existing boolean-only `verifyTotpCode()` (kept, now implemented in terms of the new function, still exported since `totp.test.js` exercises it directly).
**Why:** The prior implementation (`verifyTotpCode()` returning only a boolean) had no anti-replay tracking at all — a valid code stayed usable for its full ~90s window (current period ±1 for clock skew) even after the legitimate holder already used it, letting an attacker who intercepts a code in transit (proxy, exfiltration, shoulder-surfing) replay it. A counter, not a wall-clock timestamp, is the correct thing to persist: `otpauth`'s `TOTP.validate()` returns a `delta` (which of the ±1 window periods actually matched), and the *matched* period is what must never repeat — comparing against wall-clock time at verification time would either reject a legitimate code near a period boundary or fail to catch a replay of a code from the edge of the window, depending on which boundary. Kept `verifyTotpCode()` as a thin wrapper rather than deleting it, since breaking it would have forced a change to `totp.test.js` that carries no other benefit — its own tests aren't about anti-replay.
**How to apply:** Any endpoint accepting a TOTP code and issuing a session/granting access on success MUST use `validateTotpToken()` and persist the returned counter to the account it authenticated, comparing against the previous value on every future attempt — `verifyTotpCode()` alone is insufficient for anything session-granting, since it has no way to signal what to persist for anti-replay. `mfa-verify.js` and `mfa-enroll.js`'s confirm step are the only two call sites today; both were migrated.

**Decision:** Session 20 step 10 added audit-trail recording (`resolveAuditRecorder(deps)` + `auditRecorder.record()` after each mutation, exactly matching `ticketStore.js`'s established pattern) to 12 store files that had none: `scopeOfWorkStore.js`, `changeOrderStore.js`, `paymentRequestStore.js`, `subscriptionStore.js`, `assetStore.js`, `reminderStore.js`, `serviceRecordStore.js`, `websiteProfileStore.js`, `itSupportStore.js`, `workLogStore.js`, `approvalStore.js`, `entitlementStore.js`, `templateStore.js`. Two functions with no `organizationId` in scope (`markBackupRestoreVerified`, `updateServiceRecordStatus`) now fetch the row first, and now throw on a not-found id instead of silently no-op'ing on a zero-row UPDATE.
**Why:** A repo-wide RBAC/audit review (step 10) found this gap by grepping every store for audit calls — the same class of bug already found and fixed for `ticketStore.js`/`ticketWorkflowStore.js` in step 1, recurring because each of these 12 files was built in an earlier session (Sessions 11-19) before the `resolveAuditRecorder` convention was established, and nothing enforced it retroactively across the whole `src/db/` directory. Left unfixed, roughly a third of all Care Hub endpoints could mutate data (create/alter change orders, payment requests, subscriptions, service records, approve/reject approvals, falsify backup-verification) with zero audit trail, directly undermining `audit-log.js` — the platform's only mutation-visibility tool for platform_admin. `changeOrderStore.createChangeOrder()`'s internal call to `approvalStore.createApprovalRequest()` needed to explicitly thread its own `auditRecorder`/`actorId` through (`{ sql, now, idGenerator, auditRecorder, actorId: deps.actorId }`), not just `{ sql, now, idGenerator }` — without that, `approvalStore.js`'s now-independent audit call would fall back to its own real `createPgAuditSink({sql: deps.sql})`, causing a real (not test-fake) audit write to fire against whatever `sql` mock a test supplied, breaking call-count assertions; this was caught by the full suite, not missed silently.
**How to apply:** Any new `src/db/*Store.js` mutating function must call `resolveAuditRecorder(deps)` and record an event after every successful write, from the moment it's written — do not defer this "until an endpoint needs it," since that's exactly how these 12 files ended up unaudited for one to several sessions. When one store function calls another store's mutating function internally (as `changeOrderStore` calls `approvalStore`), always thread `auditRecorder`/`actorId` through explicitly in the inner call's `deps` — never let the inner call silently fall back to its own real sink.

**Decision:** Session 20 step 8's "fail-closed email configuration" was implemented as a security-notification-plus-audited-delivery pattern (`mfa-enroll.js`'s confirm step, `mfa-manage.js`'s disable/reset), not as making `_lib/email.js`'s `sendEmail()` itself throw when unconfigured or when a provider call fails.
**Why:** Every existing caller of `sendEmail()` (account verification, invitation links, message notifications) is written as deliberately best-effort — the whole platform is designed to work end-to-end with no email provider configured yet, with tokens/links retrievable directly from Netlify Blobs as an escape hatch (see `email.js`'s own header comment and `README_ADMIN_SETUP.md`). Making `sendEmail()` throw on failure would break that invariant for every existing caller, not just the new security-sensitive ones, and would turn a missing `RESEND_API_KEY` (an expected pre-launch state, not an error condition) into a 500 on account registration/invitations/messaging. The actual security gap step 10 flagged wasn't "sendEmail can fail" — it was "a security-relevant email can fail *silently*, with the platform having no record that a notification never reached the account owner." Fixing that means every security-critical call site checks `sendEmail()`'s return value and records its own audit event reflecting delivery outcome, which now happens for MFA enroll/disable/reset (`mfa.enroll.notification`, `mfa.disable.notification`, `mfa.reset.notification`) — `outcome: "failure"` with a `reason` when not delivered, so an operator can see the gap in `audit-log.js` rather than assuming silently that account owners were notified.
**How to apply:** Any future security-critical email (password reset, suspicious-login alert, etc.) should follow this same pattern — call `sendEmail()`, then unconditionally record an audit event keyed off its `sent` field, never assume delivery succeeded just because the call didn't throw. This does not by itself close the Critical MFA-enrollment-hijack finding (`SECURITY_REVIEW.md`) — an attacker who already has the pre-auth cookie can still complete enrollment regardless of whether the owner gets notified — but it makes the gap visible and auditable instead of silent, and gives the legitimate owner a chance to react quickly if they do receive the email.
