|---|---|---|---|

**The endpoint:** `netlify/functions/tickets.js` — `POST` (create, `request.submit`), `GET` (list, `request.view`/`ticket.view` depending on role), `PATCH` (transition, `ticket.work`, requires genuine technician assignment). Every decision — validation, transition legality, authorization — still happens in the existing pure engines (`ticketSubmission.js`, `ticketLifecycle.js`, `rbac.js`); this file only authenticates, authorizes, and translates HTTP.

**A discovered gap, documented rather than worked around:** `rbac.js`'s technician `ticket.view` check is inherently per-resource (`assigned: true` for *one* ticket), so it cannot honestly authorize "list every ticket in an org" for a technician — that's a different concern (`src/admin/workQueueViewModel.js`, F051, already built but not yet exposed as its own endpoint). `tickets.js`'s `GET` correctly denies technicians rather than faking an `assigned: true` it can't back up. Flagged as follow-up work, not silently patched over.

**A real architecture problem, solved rather than deferred:** no Netlify Function in this repository has ever been unit-testable — `@netlify/blobs` throws `MissingBlobsEnvironmentError` outside a real Netlify runtime, and `netlify dev`'s local Blobs emulation couldn't be reached this session (the auto-detected linked site returned a 404 on its addons endpoint; `--offline` avoided the crash but also skipped Blobs emulation entirely). Rather than leave the endpoint layer permanently untested, `care_hub_auth.js` and `tickets.js` were both given the same `deps = {}` override seam every `src/db/*` store already uses — `exports.handler = async (event, context, deps = {}) => {...}`. Netlify always calls handlers with exactly `(event, context)`, so `deps` defaults to the real implementations in production with zero behavior change; tests can now inject a fake session resolver while the real `rbac.js` + real Postgres store functions still run underneath. This closes one real slice of audit finding F016 ("zero automated tests... in the pre-Care-Hub repository") for the first time.
